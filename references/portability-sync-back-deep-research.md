# Patient Data Portability Deep Research Addendum

## Scope

This addendum extends the Trustcare Network portability design with two additional VC-only outbound documents and a sync-back architecture for heterogeneous HIS and legacy databases.

- Medical certificate: issued only as a Verifiable Credential, with FHIR `Composition` and `DocumentReference` evidence.
- Prescription: issued only as a Verifiable Credential, with FHIR `MedicationRequest` evidence and optional dispense sync-back.
- Sync-back: adapter-driven, idempotent, auditable writes back to HIS, FHIR servers, HL7 v2 brokers, REST APIs, CSV/SFTP batch, and legacy DB outbox tables.

## Standards Alignment

- FHIR R4 `MedicationRequest` is the canonical source for an order or request for medication supply or administration.
- FHIR R4 `Composition` is the canonical clinical document structure for attested documents such as a medical certificate.
- FHIR R4 `DocumentReference` carries document metadata, hash, attachment pointers, and provenance-friendly references.
- FHIR REST supports create/update/conditional update and version-aware interactions, which should be preferred where HIS exposes a FHIR facade.
- W3C Verifiable Credentials define issuer, holder, verifier, credential subject, proof, and credential status concepts used by Trustcare VC envelopes.
- SMART Health Links should be used as a sharing mechanism for IPS/referral packets where the patient or clinician needs QR/link portability with expiry, passcode, manifest, and encrypted payload handling.

## VC-Only Outbound Document Rule

Every document leaving Trustcare must be represented as a VC or VP.

- If the payload is a clinical summary, issue `PatientSummaryCredential`.
- If the payload is consent evidence, issue `ConsentReceiptCredential`.
- If the payload is a medical certificate, issue `MedicalCertificateCredential`.
- If the payload is a prescription, issue `PrescriptionCredential`.
- If the payload is a claim submission package, issue `ClaimPackageCredential`.
- If the payload is evidence of a successful write back to a legacy system, issue `SyncReceiptCredential`.

The VC should contain a minimal human/verifier claim set plus hashes/references to canonical FHIR evidence. Sensitive clinical data can be minimized through selective disclosure or SHL-backed retrieval. The current implementation signs JWT VC/VP envelopes with `jose` for development and CI; production should replace the signing secret with KMS/HSM and can upgrade the envelope to SD-JWT VC without changing the FHIR evidence model.

## Medical Certificate VC

FHIR evidence:

- `Composition`: certificate title, authoring practitioner, patient subject, issuing organization, clinical statement, fitness status, recommendations, validity period.
- `DocumentReference`: points to the certificate artifact or FHIR JSON representation and includes hash/content type.
- `Provenance`: generated by the canonical layer when the certificate is derived from HIS facts.
- `AuditEvent`: records issuance, verifier presentation, revocation, and access.

Credential subject fields:

- patient id/name
- practitioner id/name/license
- organization id/name
- issued date
- valid from/until
- diagnosis statement, only if policy permits
- fitness for work/travel status
- recommendations
- FHIR composition hash and embedded/reference evidence

Operational rules:

- Issue only after practitioner authorization.
- Use short validity windows by default.
- Revoke when issued in error or when the legal basis changes.
- For cross-border/medical tourist flows, pair with SHL/IPS where the receiving party needs clinical backing data.

## Prescription VC

FHIR evidence:

- `MedicationRequest`: one resource per prescribed medication.
- `MedicationDispense`: optional sync-back from pharmacy/HIS after dispensing.
- `Provenance`: source and authoring evidence.
- `AuditEvent`: issuance, presentation, dispense, cancellation, and sync-back events.

Credential subject fields:

- patient id/name
- prescriber id/name/license
- issuing organization
- authored date
- dispense window
- substitution allowance
- repeat allowance
- medication requests and bundle hash

Operational rules:

- Pharmacy verifier checks issuer, expiry, revocation, prescriber, medication request hash, and patient-holder binding.
- Dispense completion should generate a sync-back plan and, after acceptance, a `SyncReceiptCredential`.
- Refill/repeat logic must be enforced by credential status and target HIS readback, not by client UI alone.

## Sync-Back Architecture

Trustcare must not assume that it owns the system of record. Each adapter declares its write mode:

- `system_of_record`: target HIS accepts authoritative writes.
- `system_of_reference`: target can receive references or review items, but not automatic clinical updates.
- `mirror_only`: target receives a copy/outbox event for reconciliation and reporting.

Supported adapter kinds:

- `fhir_rest`: preferred when a HIS exposes FHIR R4 endpoints. Use conditional create/update, ETag/version preconditions, Bundle transactions, and readback verification.
- `hl7v2`: send event messages such as ADT/ORU/RDE/MDM and require ACK plus scheduled reconciliation because optimistic versioning is usually absent.
- `db_view`: write to a dedicated outbox/staging table instead of mutating vendor tables directly.
- `rest_api`: use idempotency keys and target-specific version/checksum fields.
- `csv_batch`: generate signed batch files with manifest hashes and import receipts.
- `manual_queue`: route to a human reconciliation queue where automatic write-back is unsafe.

## Consistency Controls

Every sync-back plan includes:

- idempotency key
- source event id
- patient business key
- payload hash
- consistency key
- expected target version when available
- preconditions
- rollback/reconciliation hint
- audit event

Consistency strategy by capability:

- Strong consistency: FHIR REST or transactional DB outbox with version check and readback.
- Eventual consistency: HL7 v2, REST APIs without versioning, CSV/SFTP imports.
- Manual consistency: unsafe legacy systems, ambiguous identity matches, or clinical conflicts.

Required post-write steps:

1. Persist outbound sync plan.
2. Send to target adapter.
3. Capture ACK/import result.
4. Read back target resource or staging receipt.
5. Compare target version/checksum/business key.
6. Issue `SyncReceiptCredential` only after accepted or reviewed.
7. Schedule reconciliation for non-transactional targets.

## Implementation Notes

The current PR adds:

- `server/portability/fhir.ts`: HIS/HL7/FHIR-native input to canonical FHIR R4 Bundle with Provenance.
- `server/portability/vc.ts`: signed VC/VP issuance and verification.
- `server/portability/clinicalDocuments.ts`: medical certificate and prescription FHIR evidence builders.
- `server/portability/policy.ts`: purpose-based consent/data-minimization decisions.
- `server/portability/syncBack.ts`: sync-back target definitions, consistency plan generation, deterministic adapter execution boundary, ACK/readback modeling, and receipt resources.
- `server/portability/trust.ts` and `server/portability/vc.ts`: production verifier policy, trust registry inputs, asymmetric JWK/JWKS support, and status-list revocation checks.
- `portability` tRPC router and `/portability` UI workbench.
- `references/production-hardening-runbook.md`: deployment checklist for signing keys, JWKS, trust registry, revocation, and reconciliation jobs.

The implementation now closes the sandbox loop from plan to deterministic execution to `SyncReceiptCredential`, so CI/E2E can validate VC issuance and consistency checks without connecting to real hospital systems. The next deployment hardening step is replacing deterministic adapter execution with site-specific workers for each HIS, HL7 broker, and legacy database target, while preserving the same idempotency, ACK, readback, reconciliation, and receipt VC contract.
